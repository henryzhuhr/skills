# 🏷️ Version Tag Helper

用这个 skill 以安全、可重复的方式给 Git 分支打标签。

## 标准工作流程

### 1. 先检查当前状态

1. 执行命令 `git branch --show-current` 查看当前分支。
2. 执行命令 `git tag --list '<pattern>'` 查看已有标签。
3. 执行命令 `git describe --tags --abbrev=0` 查看最近的标签。
4. 询问用户想要执行的操作类型。

### 2. 选择操作类型

提供以下选项供用户选择：

```bash
请问你想要做什么？

1. 给当前分支打标签 - 基于版本号或自动生成
2. 给指定分支打标签 - 告诉我分支名
3. 查看标签列表 - 支持按模式筛选
4. 删除标签 - 告诉我标签名
5. 推送标签到远程 - 单个或全部
```

### 3. 打标签操作

#### 轻量标签（默认）

```bash
git tag <tag-name>
```

#### 附注标签（推荐用于版本发布）

```bash
git tag -a <tag-name> -m "release: <description>"
```

#### 基于语义化版本

- 支持 `v1.0.0`、`v1.0.0-alpha`、`v1.0.0-beta.1`、`v1.0.0-rc.1` 等格式
- 如果用户未指定前缀，自动添加 `v` 前缀
- 支持自动递增版本号（基于最近的 tag）

### 4. 验证结果

- 执行 `git tag --list <tag-name>` 确认标签已创建。
- 执行 `git show <tag-name>` 查看标签详情。
- 如果是附注标签，显示标签信息（tagger、日期、消息）。

### 5. 推送到远程（可选）

- 单个标签：`git push origin <tag-name>`
- 全部标签：`git push origin --tags`
- 推送前必须明确询问用户确认。

## 安全规则

- 打标签前检查标签是否已存在，避免覆盖。
- 如果标签已存在，询问用户是否删除后重建或直接报错。
- 默认不给其他分支打标签，除非用户明确指定。
- 推送标签前必须获得用户确认。
- 不自动推送，除非用户明确要求。

## 执行要求

- 优先使用标准 Git 命令，不依赖额外脚本。
- 标签命名遵循语义化版本规范（SemVer）。
- 标签不存在：成功创建后汇报标签名和对应的 commit hash。
- 标签已存在：属于失败，停止执行并暴露错误，等待人工处理。
- 分支不存在、命令执行失败：属于失败，直接停止并暴露错误。
- 汇报结果时明确区分：已创建的标签、因冲突未创建的标签。

## 标签命名规范

- 版本号以 `v` 开头，如 `v1.0.0`
- 支持预发布版本：`v1.0.0-alpha.1`、`v1.0.0-beta.2`、`v1.0.0-rc.1`
- 支持构建元数据：`v1.0.0+build.123`
